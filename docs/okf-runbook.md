# OKF runbook profile — modeling how a system is run, operated, and observed

> **Status:** P1 landed (types + generation); P2–P4 proposed. The three ostler types (§4, §9)
> and the build-pass generation (§5) ship; the live-walkthrough consumption + reconciliation
> (§6–§7, phases P2–P3) are not yet implemented. This document specifies an extension to the
> [OKF UI profile](okf-ui-profile.md): three new node types (`environment`, `runbook`,
> `step`) that make a repo's **operational surface** — how you stand it up, in which
> environment, and how you observe it — first-class, machine-readable OKF. It also
> specifies how the **okf-builder** workflow *generates* these nodes from code (first
> pass) and *consumes* them to boot and reconcile the app (second pass). Where this spec
> and shipped tooling differ once implemented, the shipped behavior wins and is called out
> inline. This profile is **generic and repo-agnostic**; the only repo named here is
> `groom` (native to this repo). All other examples are neutral shapes.

## 1. Why this profile exists

The [UI profile](okf-ui-profile.md) models *what a system does* — screens, commands,
endpoints, concepts. It has no node for *how you run the system*: the launch command, the
services it depends on, the ports and health checks, the environments (local/test/prod)
and their URLs, the dev CLI. Today that knowledge lives outside the book — in Makefiles,
compose files, `package.json` scripts, config loaders, and README prose — so:

- a reader of the book cannot tell how to stand the system up or reach it;
- **okf-builder's live walkthrough cannot boot the app it is supposed to walk** — it has
  no machine-readable launch recipe, so it either boots the wrong single process or skips
  discovery entirely;
- there is no coverage pressure forcing the operational surface to be documented at all.

groom is the illustrative case: it has a rich book (a `server`, a `screen`, three `cli`
nodes, dozens of `concept`s), yet its launch/entry/health facts are *scattered* across
`screen.route: /`, the server's startup-hook prose, and the CLI's `usage:` — with no
single node that says "run `groom serve`, it binds `:8787`, it is ready when the port is
bound." This profile gives that fact a home.

### Goal — the operational surface is derived, not hand-authored

A `runbook` is **generated by okf-builder from the repo's own operational evidence**
(§5), the same way every other node is derived from code. It is authored to be complete
enough that a competent agent or a script can **stand the system up and observe it** from
the doc alone. The runbook is the **interface between okf-builder's two passes**: the
build pass derives it from static evidence; the walkthrough pass boots from it and
reconciles it against what actually comes up (§7).

### Non-goals

- Not a deployment/orchestration engine. A runbook *describes* the recipe; it does not
  replace compose, Pulumi, or CI. The `iac` driver (§4.1) documents a deploy definition; it
  does not run it.
- Not a per-repo hardcode. The types and the generator are generic; every concrete port,
  command, and URL lives in a repo's own book, never in this spec or the shared workflow.
- Not JSON-Schema-validated. Like the rest of OKF, the format stays loose; the linter
  checks conformance + link integrity, and spec-completeness (§4.4) is a review standard.

## 2. Design principles

1. **The observable surface generalizes beyond "a service with a URL."** A system's
   observable surface may be a browser page, a mobile UI reached over a debug bridge, an
   HTTP API, a stream of stdout, or a set of **output files on disk**. The profile must fit
   a tool that stands up *nothing* (a batch generator whose only output is artifacts) as
   naturally as a multi-service web stack.
2. **Boot is an ordered pipeline where "stand something up" is optional.** The recipe is a
   sequence of typed steps (§4.3). Some repos have one long-running service; some have a
   fabric of them; some have none.
3. **Readiness is polymorphic.** "Ready" may be an HTTP 200 on a real endpoint, a
   WebSocket frame, a bound port, a log line, a passing semantic health command, or the
   existence + golden-verification of output artifacts. A readiness contract that assumes
   "poll a URL for 200" is both too narrow and a false-positive trap (an SPA shell returns
   200 with a dead backend).
4. **One repo has 1..N surfaces, each with its own driver.** A monorepo may expose a
   browser dashboard, a mobile app, an HTTP-only service, and a batch CLI at once. The
   profile models multiple **single-driver** runbooks sharing environments rather than one
   overloaded node.
5. **Generic base, repo-owned content.** The types and generator carry no repo names,
   ports, or commands. Everything concrete is authored into a repo's book by the generator.
6. **Ostler parses and gates on shape.** As with the UI profile, ostler recognizes the
   three types, resolves their links, and `list`/`search`/`trace` them; structural rules
   are `doctor` gates with a mechanical `fmt`/`scaffold` remedy.

## 3. Relationship to the UI profile

The three new types are added to the UI-profile node vocabulary
([okf-ui-profile.md §3](okf-ui-profile.md)) as ordinary built-in types in the ostler
registry — `runbook` and `environment` are **file-level**, `step` is **section-level**
(a `### <id>` under a `## Steps` heading), exactly like `command` under `## Commands`. They
are recognized, listed, and linted with no bundled JSON Schema, like `spec.*`. A runbook's
`surfaces:` links reuse the existing surface types (`screen` / `server` / `cli` / `format`)
— there is **no** new "surface" type.

## 4. Node types

### 4.1 `runbook` (file-level) — one single-driver recipe

How to bring one **driver's** view of the system up and observe it, in a given
environment. A repo has one runbook per driver it exposes.

Bullets (canonical order):
- `driver:` — the observation/drive mechanism this runbook serves. Fixed vocabulary:
  - `web` — a browser page; observed by navigating an entry URL.
  - `mobile` — a mobile app on an emulator/simulator/device; observed via a debug bridge
    and/or a UI-driver tool.
  - `http` — an API with no UI; observed by request + response (health, introspection).
  - `cli` — a bounded command; observed via stdout + exit code.
  - `artifact` — a batch job; observed via produced output files + verification.
  - `iac` — a deploy definition; documentation-only, never run by the walkthrough.
  - `none` — not driven; documentation-only.
- `environment:` — link to the target `environment` node this runbook boots (default: the
  repo's `local` environment).
- `cli:` — link to the dev-CLI `cli` node this runbook drives with, when one exists.
- `surfaces:` — links to the `screen` / `server` / `cli` / `format` nodes this runbook
  exposes (what a driver will observe).
- `code:` — the launch entry point (`path::symbol`), when there is one.

Required section: `## Steps` — an ordered list of `### <id>` `step` nodes (§4.3).

### 4.2 `environment` (file-level) — one target

A place the system runs. A repo has 0..N (a purely local tool may have none; most have
`local` and `prod`).

Bullets:
- `selector:` — how this environment is chosen (e.g. a stage env-var value, an env-file).
- `services:` — nested: one child per service, giving its URL/host **in this environment**
  (ports and hosts are environment-scoped — they may differ local vs prod). A child may
  carry a **host-rewrite** note (e.g. a mobile emulator loopback alias, or a LAN IP for a
  physical device) with its reason.
- `backing:` — nested: backing projects, databases, buckets, emulators this environment uses.
- `local-only:` — `true` when tooling must refuse to target this environment without an
  explicit override (a safety gate orthogonal to which environment it is).

### 4.3 `step` (section-level, under `## Steps`) — one ordered boot step

Bullets:
- `kind:` — fixed vocabulary:
  - `prepare` — a blocking setup command (install/build/deps/stack-up); run to completion.
  - `service` — a long-running process; launched, health-gated, reaped on teardown.
  - `seed` — populate observable state (fixtures) so there is something to observe.
  - `run` — a bounded, artifact-producing job (the `artifact` driver's core step).
  - `health` — a readiness check (an endpoint, a semantic health command, a WS frame, …).
  - `verify` — confirm success without a live probe (golden/deterministic/assertion).
  - `drive` — informational: how the driver reaches/operates the surface this step exposes.
- `run:` — the exact bounded command.
- `working-directory:` — cwd, when not the repo root.
- `env:` — nested env-var wiring this step needs.
- `health:` — for `service`/`health` steps, the readiness signal: a URL to poll, a command
  to run, a `port-bound` / `log:<pattern>` / `ws:<frame>` marker. **Must be a real
  readiness signal, never a shell that is served with the backend down.**
- `produces:` — for `run` steps, the output artifact path(s)/glob(s) — the observable surface.
- `verify:` — for `run`/`verify` steps, how success is confirmed (a golden compare, a
  deterministic re-run, an assertion string, a test id).
- `optional:` — `true` for best-effort steps (a secondary service, a device that may be absent).
- `depends-on:` — ordering hint (default: document order).
- `provenance:` — `derived` (written by the build pass from static evidence) or `verified`
  (promoted by the walkthrough after it booted successfully — §7).

### 4.4 Spec-completeness bar

A `runbook` is at-bar when, from the doc alone, a reader can stand the system up and reach
its surface: the `## Steps` are ordered and each is executable (a real `run:`, a real
readiness `health:`/`verify:`), the `environment` it targets resolves, the `driver` is
correct, and `surfaces:` names what will be observed. An `environment` is at-bar when its
`selector`, per-service URLs (env-scoped, with any host-rewrite reason), backing, and
`local-only` safety are complete. As elsewhere in OKF this is a **review** standard, not a
`doctor` gate.

## 5. Generation — the build pass derives the runbook

The runbook is produced by okf-builder's existing build machinery, at four hook points.
The **forcing function** is coverage: today `inventory-source` inventories only code
symbols, so the operational surface is invisible and nothing pressures a runbook to exist.

1. **`enumerate-surfaces`** — in addition to code entry points, detect the **operational
   surface** from generic evidence: `Makefile`/`justfile` targets, `docker-compose*`
   services, `package.json` scripts, `pyproject` console-scripts / `__main__` modules,
   `Dockerfile`/CI, config-loader stages + env vars, and README "how to run" sections. Emit
   one `runbook` seed per driver and one `environment` seed per target.
2. **`investigate`** — document a `runbook` / `environment` / `step` item to the §4.4 bar
   from that static evidence: resolve the ordered steps, ports, health signals, env wiring,
   and driver classification. Mark every step `provenance: derived`.
3. **`inventory-source`** — additionally materialize an **operational-surface inventory**
   (the run targets/services/scripts/entry points above) so an undocumented run surface is a
   coverage unit.
4. **`recheck-coverage`** — judge `runbook`/`environment` coverage against that inventory
   and queue what is missing. **The book is not complete until the run surface is a runbook.**

Because generation is evidence-based, it is uniform across repo shapes: a web app yields a
`web` runbook; a batch tool yields an `artifact` runbook whose steps are `prepare → run →
verify`; a multi-surface monorepo yields several single-driver runbooks sharing environments.

## 6. Consumption — the walkthrough pass boots from the runbook

A **driver-specific** walkthrough (`walkthrough-web` today; a `walkthrough-mobile` or a
`produce-verify` consumer later) selects the runbooks whose `driver` matches it and boots
them through **one generic step engine**:

- `prepare` / `seed` / `health` / `verify` / `run` steps run to completion; a non-zero exit
  aborts the boot unless the step is `optional`.
- `service` steps launch long-running (own process group), are health-gated on their real
  `health` signal, and their process-group ids are appended to a **persisted pgids file** so
  teardown reaps every service even if a run crashes mid-boot (and a stale pgids file from a
  prior crash is reaped at next boot).
- On any non-optional failure the engine reaps everything it started (reverse order) and
  reports failure.

**The driver rule that keeps this general:** a driver walkthrough **hard-aborts** (a
`fail` terminal, non-zero exit) only when a runbook *for its own driver* will not boot; it
**soft-skips** cleanly when the repo has no runbook for that driver. So a batch/artifact
repo under `walkthrough-web` is correctly skipped, not forced into a browser walk, while a
web repo whose stack won't come up fails loudly instead of silently skipping discovery.

## 7. Reconciliation — derived vs verified, never wipe

The walkthrough treats the generated runbook as the base and **reconciles** it:

- Steps it successfully executes are promoted `derived → verified`; ports/health/env it
  observes to differ from the derived values are corrected in place.
- **Unobserved ≠ absent.** A step, service, or surface the walk did not reach this run is
  left intact — never deleted or contradicted. Healing is additive by default; a bullet is
  overwritten only when actively re-observed.
- A turn that could not satisfy a precondition (the stack did not come up, a gate redirected
  it) must **not** write negative conclusions ("not reachable") into the book — that is a
  property of the broken precondition, not the product.

This mirrors the UI profile's "book is the current spec, story is the delta" rule
([okf-ui-profile.md §2](okf-ui-profile.md)), extended to the operational surface: the
runbook is merged toward current reality, never replaced wholesale or wiped.

## 8. Worked shapes (neutral)

The schema is validated to fill, without force-fitting, across a deliberately diverse set:

- **A browser web app** — `environment` {local, test, prod}; one `web` runbook; steps
  `prepare` (stack-up) → `service` gateway (health = a real API endpoint) → `service` web
  (env wires the auth emulator; health = the shell). Surfaces = `screen` nodes.
- **A mobile app** — one `local` environment (services carry a host-rewrite for the
  emulator/device); one `mobile` runbook; steps bring up the emulator fabric, the API, a
  `seed` step, then a `drive` step over the debug bridge / UI-driver tool. Dev CLI is a
  `cli` node.
- **groom** — one `local` environment (a trust/bind knob, `local-only`); one `web` runbook;
  steps `prepare` (`uv sync`) → `service` (`groom serve`, `health: port-bound` / a WS
  frame — there is no `/health`) → `seed` (inject state so the dashboard is not empty).
  Surface = its one `screen`.
- **A batch/artifact tool** — no environment (or a trivial one); one `artifact` runbook;
  steps `prepare` → `run` (`produces:` the output files) → `verify` (golden / deterministic /
  an assertion string). Surfaces = `format` and `cli` nodes. It has nothing to boot and no
  URL — the `web` driver soft-skips it.
- **A multi-surface monorepo** — `environment` {local, prod} with env-scoped ports; several
  single-driver runbooks (`web` dashboard, `mobile` app with a LAN-IP host rewrite, `http`
  side service, `cli` batch job) sharing the environments; a prod/local env-file toggle
  recorded as a `local-only` safety note.

## 9. Ostler change surface

The type additions are registry-driven and small ([ostler registry](../ostler/ostler/registry.py)):
add `environment`, `runbook` (file-level), and `step` (section-level, heading `Steps`) to
`UI_TYPES`; update the vocabulary table in [okf-ui-profile.md §3](okf-ui-profile.md); add the
names to the `test_ui_profile.py` type-set assertion; add scaffold/list tests. Formatting,
doctor validation, scaffold, list, and search pick the types up automatically. A built-in
type is available to every repo with no per-repo config.

## 10. Phasing

- **P1 — types + generation:** the three ostler types; build-pass generation (enumerate the
  run surface → investigate → operational-surface inventory → coverage). Every repo's build
  produces `runbook`/`environment` nodes with `derived` steps.
- **P2 — boot + hard-abort:** the driver-generic step engine; `fail` on a same-driver boot
  failure; persisted pgids + crash-safe teardown; the driver soft-skip rule; a fresh browser
  session per run.
- **P3 — reconcile robustness:** `derived → verified` promotion + observed-value correction;
  the unobserved-≠-absent / never-wipe discipline; the precondition-blocked write guard;
  cross-run staleness GC; tests.
- **P4 — breadth:** future `walkthrough-mobile` / `produce-verify` driver consumers; validate
  generation by running the builder across the diverse shapes of §8.

## 11. Open questions

1. Granularity: three types (`environment` + `runbook` + `step`) vs. one fatter `runbook`
   carrying environments and steps as nested bullets. This spec takes the three-type option
   (each stays small; `step`/`environment` get validation + queryability).
2. Multi-driver repos: multiple single-driver runbooks (this spec) vs. one multi-surface
   runbook. The former keeps each runbook simple for its single driver consumer.
3. Whether `run`/`verify` for the `artifact` driver warrants its own future
   `produce-verify` walkthrough consumer, or stays documentation-only for now.
