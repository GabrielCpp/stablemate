# Operator browses workspace file

This journey covers the as-built operator path through the [groom dashboard](../gui/screens/groom-dashboard.md) Files pane: activating [files mode](../gui/screens/groom-dashboard.md#select-activity-files-mode), opening the [files repository picker](../gui/screens/groom-dashboard.md#open-files-repository-picker), selecting a [repository menu option](../gui/screens/groom-dashboard.md#select-repository-menu-option), loading [workspace file list data](../workspace-file-list-data.md) through [GET /files/{container_id}](../http/groom.md#get-workspace-file-list), selecting a [files file row](../gui/screens/groom-dashboard.md#select-files-file-row), loading [workspace file content data](../workspace-file-content-data.md) through [GET /file/{container_id}](../http/groom.md#get-workspace-file-content), and rendering the selected file through the [dashboard file view renderer](../concepts/dashboard-file-view-renderer.md). The server-side reads prefer [sidecar live sessions](../sidecar-live-sessions.md) RPCs and fall back to the workflow workspace volume readers when no live sidecar can serve the data.

- start: the groom server is running, the browser has loaded the [groom dashboard](../gui/screens/groom-dashboard.md), and at least one [workflow container](../concepts/workflow-container.md) may have a known `workspace_volume`. The dashboard may start in inbox mode with no selected repository; an existing selected repository is allowed and is reused when the operator enters files mode.
- steps:
  1. The operator activates [select activity files mode](../gui/screens/groom-dashboard.md#select-activity-files-mode) from the activity bar. The dashboard sets `.app[data-mode]` to `files`, marks the Files activity control active, closes the repository menu if it was open, and enters the Files pane loader.
  2. If no repository is selected, [select activity files mode](../gui/screens/groom-dashboard.md#select-activity-files-mode) renders `Pick a container / repo above.` in `#files-tree` and does not issue a file-list request. If a repository is already selected, it immediately resets `#file-view` to `Select a file to view it.`, requests [GET /files/{container_id}](../http/groom.md#get-workspace-file-list) with the selected `repo`, and renders the returned path list as files-tree rows, `(no files)`, or `failed to load`.
  3. The operator activates [open files repository picker](../gui/screens/groom-dashboard.md#open-files-repository-picker) from the native Files pane repository button. The browser positions and opens the shared repository menu, replaces `#repo-menu` with `Loading...`, clears and focuses [repository menu search input](../gui/screens/groom-dashboard.md#repository-menu-search-input), and fetches [GET /repos](../http/groom.md#get-repository-menu).
  4. [Serve repository menu](../http/groom.md#serve-repository-menu) reads the current in-memory workflow registry, keeps workflows with known workspace volumes, resolves checkout directories through the [workspace volume repository-directory reader](../concepts/workspace-volume-repository-directory-reader.md), and returns [repository menu data](../repository-menu-data.md) rendered as one [repository menu option](../gui/screens/groom-dashboard.md#repository-menu-option) per eligible container/repository pair. Checkout-discovery failures for one workflow become that workflow's root option or no checkout entries, not a failed endpoint response.
  5. The dashboard inserts the repository option HTML into `#repo-menu`. The operator may narrow visible options through [filter repository menu options](../gui/screens/groom-dashboard.md#filter-repository-menu-options), which hides or shows already-loaded option rows by case-insensitive `data-label` matching without requesting new data.
  6. The operator activates [select repository menu option](../gui/screens/groom-dashboard.md#select-repository-menu-option). The dashboard stores [dashboard selected repository state](../dashboard-selected-repository-state.md) from the option's `data-container`, `data-repo`, and `data-label`, updates every repository picker label, closes the menu, and dispatches the active-pane loader for the current files mode.
  7. The active-pane loader calls the Files pane load path. It resets `#files-tree` to `Loading files...`, resets `#file-view` to `Select a file to view it.`, and sends [GET /files/{container_id}](../http/groom.md#get-workspace-file-list) with the selected container id as a route segment and the selected repository path as the `repo` query value.
  8. [Serve workspace file list](../http/groom.md#serve-workspace-file-list) first asks the connected sidecar, when present, for `getTree` with the selected `repo`. A successful sidecar response supplies [workspace file list data](../workspace-file-list-data.md) as `paths` and is serialized to newline-separated `text/plain`.
  9. If the sidecar is absent, times out, closes, or reports an error, [serve workspace file list](../http/groom.md#serve-workspace-file-list) falls back to the workflow's known workspace volume and reads repo-relative file paths through the [workspace volume file-list reader](../concepts/workspace-volume-file-list-reader.md). Missing workflow state, missing volume metadata, non-zero fallback reads, or an empty selected checkout produce an empty plain-text response rather than an endpoint-specific error body.
  10. The dashboard reads the file-list response as text, splits on newline, trims each line, discards blanks, and builds a [dashboard files path tree](../dashboard-files-path-tree.md). It then renders [files directory toggle](../gui/screens/groom-dashboard.md#files-directory-toggle) rows for directory nodes and [files file row](../gui/screens/groom-dashboard.md#files-file-row) rows for file leaves. With no normalized paths it renders `(no files)`; on fetch or body-read rejection it renders `failed to load`.
  11. The operator may expand or collapse directories through [toggle files directory](../gui/screens/groom-dashboard.md#toggle-files-directory). This changes only local DOM visibility for that directory branch, does not fetch data, and does not change the selected repository or file viewer.
  12. The operator activates [select files file row](../gui/screens/groom-dashboard.md#select-files-file-row). The dashboard makes the clicked row the only visually active file row, reads its `data-path`, sets `#file-view` to `Loading...`, and sends [GET /file/{container_id}](../http/groom.md#get-workspace-file-content) with selected `repo` and selected file `path` query values.
  13. [Serve workspace file content](../http/groom.md#serve-workspace-file-content) first asks the connected sidecar, when present, for `getFile` with `repo` and `path`. A successful sidecar response supplies [workspace file content data](../workspace-file-content-data.md) as `content` and is serialized directly as `text/plain` raw file text.
  14. If the sidecar is absent, times out, closes, rejects the path, or reports an error, [serve workspace file content](../http/groom.md#serve-workspace-file-content) falls back to the workflow workspace volume. It composes `repo/path`, strips only leading slashes, guards the relative path through the [workspace volume relative path guard](../concepts/workspace-volume-relative-path-guard.md), and reads the selected file through the [workspace volume file-content reader](../concepts/workspace-volume-file-content-reader.md). Missing workflow state, missing volume metadata, empty path, unsafe path, missing file, unreadable file, or falsey content produce an empty `200 OK` text response.
  15. The dashboard passes the fulfilled response body and original selected path to the [dashboard file view renderer](../concepts/dashboard-file-view-renderer.md). The renderer replaces `#file-view` with a file header containing the escaped selected path and either `(empty or binary file)` for falsey text or a `<pre class="file-pre hljs"><code>...</code></pre>` body whose content is inserted as text.
  16. For non-empty file text, the file view chooses a Highlight.js language from the selected file extension when a known mapping exists and the loaded highlighter recognizes it. Highlighting failures are swallowed so the plain text code block remains visible; the dashboard does not render server-returned file text as HTML, mutate server state, navigate, or send websocket messages during file viewing.
- end: the Files pane remains active with the selected repository label visible, the file tree still populated from the latest file-list response, exactly one clicked file row visually marked `active` in the current tree, and `#file-view` showing either the escaped selected path with highlighted/plain raw text, the selected path with `(empty or binary file)`, or `failed to load` after a client-side fetch failure. Server-side sidecar or fallback read failures are represented as empty file-list/content data, not endpoint-specific error responses.
